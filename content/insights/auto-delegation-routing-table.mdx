## なぜキーワードルーティングが必要か

[Task ツール強制委任](/insights/task-forced-delegation) で、メインセッションが専門領域を自分で実行することを禁止しました。では、どの専門エージェントに委任すればいいか？

答えは「**キーワード検出で自動ルーティング**」。ユーザーが「EC クライアント の広告どう？」と言えば `campaign-analyst` へ、「クライアント A の予実を」と言えば `fpna-architect` へ、自動で振り分けます。

## ルーティングテーブル設計の 3 原則

<Principle num="01" heading="カテゴリ分類は 「検索意図」 ベース" />

エージェント名ではなく、**ユーザーが何について話しているか** で分類する:

- 「広告」「Google Ads」「ROAS」「CPA」「キャンペーン」 → カテゴリ: 広告・マーケ
- 「予実」「月次締め」「取締役会」「FP&A」→ カテゴリ: 経営管理
- 「SEO」「AIO」「構造化データ」「Core Update」→ カテゴリ: SEO / AIO

1 ユーザー発言に複数キーワードが含まれる場合は、最頻出カテゴリを選ぶ。

<Principle num="02" heading="カテゴリ → 専門エージェント" />

カテゴリごとに 1 つの「起点エージェント」を決める:

| カテゴリ | 起点エージェント | 用途 |
|---------|-----------------|------|
| 広告・マーケ | `campaign-analyst` | 単一エージェント分析 |
| 〈案件名〉の振り返り | `project-owner-{案件}` | PO 配下で複数 Worker 起動 |
| 予算配分・広告戦略 | `growth-head` | growth-strategy チーム起動 |
| 計測・GTM・CV 不整合 | `data-quality-checker` + `campaign-analyst` 並行 | 計測診断 |
| クライアント A・国内大手IT企業・取締役会 | `fpna-architect` | 経営管理専門 |
| 投資・DD・バリュエーション | `deal-sourcer` | 投資チーム起動 |
| 契約・NDA | `global-counsel` | 法務専門 |
| クライアントヘルス | `health-monitor` | CS チーム起動 |
| SEO / AIO / AEO | `seo-orchestrator` | SEO 統括 |

起点エージェントは **Head or Lead レベル** (Sonnet + Task tool) で、配下のサブエージェントへ再委任する責任を持つ。

<Principle num="03" heading="例外を明示化" />

ルーティングを使わない例外:

- **単一事実確認**: 「context.md 読んで」「ファイル一覧見せて」
- **既取得データの再表示**: 「さっきの数字もう一回」
- **CEO との壁打ち**: 戦略相談、判断補助
- **メタ作業**: ハーネス改修、ルール追加
- **ドキュメント更新**: context.md 更新、memory 記録

これ以外は **判断に迷ったら必ず委任**。「自分でやった方が早い」は誤り。

## 1 年間の調整履歴

テーブルは最初から完成品ではなく、**運用しながら過不足を調整** してきました。

### 追加キーワード例

- 「データ汚染」→ 当初 `campaign-analyst` だったが、計測起因が多いので `data-quality-checker` 並行に変更
- 「経営管理マスタ」→ 国内大手IT企業のライブ事業部の内部用語、`fpna-architect` に追加
- 「LLMO」→ 2026-03 に SEO 領域で頻出し始めたので `seo-orchestrator` に追加

### 削除キーワード例

- 「レポート」単独 → 汎用すぎて他カテゴリにマッチしすぎる。"週次レポート" "月次レポート" のように修飾詞と組み合わせる形に変更

### キーワード衝突の解決

複数カテゴリにマッチする場合の優先順位:

1. **プロジェクト名が含まれる場合** → そのプロジェクトの PO (`project-owner-\{案件\}` 等)
2. **2 領域に跨る場合** → 上位 Lead に委任 (`cos-chief-of-staff` など)
3. **単一カテゴリ** → テーブルで決めた起点エージェント

## 実装 — ハーネスレベルか、プロンプトレベルか

### 選択肢 A: プロンプトで指示

`CLAUDE.md` や `.claude/rules/auto-delegation.md` に「このキーワードを見たら Task ツールで委任せよ」と書く。

- 長所: 実装コストゼロ、即効性
- 短所: メインセッションが忘れることがある (Calibration 不備時)

### 選択肢 B: ハーネスの hook

Claude Code の settings.json hook で、ユーザー発言に特定キーワードが含まれているのに Task ツールが呼ばれなかったら警告を出す。

- 長所: 確実に発火
- 短所: hook の設定・メンテナンスコスト

mixednuts は現在 **選択肢 A** で運用中。10 件に 1 件くらい委任漏れが起きるが、CEO が気づいたら「委任して」と指示することで即修正。完璧を目指さず、90% を自動化する設計。

## テーブルの育て方

新しいキーワードが出てきたら:

1. 1 週間観察する (1 回だけか、継続的か)
2. 継続的なら、既存カテゴリに入るかチェック
3. 入るなら既存エージェントに追加、入らないなら新カテゴリを検討
4. 新カテゴリは慎重に (小さく始めて、2 週間後に評価)

テーブルは**最小限に保つ**。キーワードを増やすほど衝突が増えて、判定ミスが多発する。

## FAQ

**Q. このテーブルを他組織で流用できるか？**
A. カテゴリ (広告 / FP&A / 投資 / 法務) は業種を問わず共通。キーワードは自組織に合わせて調整必要。特に「プロジェクト名」カテゴリは完全に個別。

**Q. LLM が発言全体を意味解釈するのではなく、なぜキーワード検出？**
A. 意味解釈は LLM のセッションコンテキストに依存し、セッションをまたぐと判定がぶれる。キーワード検出は **明示的・再現可能・デバッグ可能**。LLM の判断余地を最小限にすることで信頼性が上がる。

**Q. キーワード検出で false positive (過剰委任) は？**
A. ある。例: CEO が「広告 の話じゃなくて、広告代理店での経験について」と言っても「広告」にヒットして campaign-analyst へ委任される。回避: 発話の context を見て、CEO が「今は違う文脈」と明示したら委任取消。

**Q. スキル (`/skill-name`) との使い分けは？**
A. スキルは "ユーザーが明示的に呼ぶ"、ルーティングは "自動で適切なエージェントを選ぶ"。スキルが呼ばれた場合はそのスキル定義に従う。呼ばれなかった場合にルーティングテーブルで判断。

**Q. 80+ キーワードで足りるか？**
A. 現時点で足りている。ただし新しい案件・新しい技術領域が増えるたびに追加。3 ヶ月に 1 回、テーブルの見直しを session-close skill でリマインドしている。
