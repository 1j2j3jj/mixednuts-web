## なぜ FP&A は遅いのか

<Answer>
月次決算は締まっているのに経営会議まで 2 週間かかる主因は "書く作業" と "判断する作業" が混在していること。経営企画が 1 人で数字の取得・集計・差異分析・文章化・経営示唆まで全部やるから遅い。AI で "書く" を代替すれば、人間は "判断" だけに集中でき、リードタイムは 1/7 に縮む。
</Answer>

事業会社の経営企画でよく見かける構造:

- **Day 0 (月末)**: 会計システムで月次締め
- **Day 1-3**: 子会社・事業部からの個別 PL 集約、連結調整
- **Day 4-6**: 予実差異の数字確認、異常値の洗い出し
- **Day 7-10**: 差異要因分析、コメンタリー (文章) 執筆
- **Day 11-13**: 上司レビュー → 書き直し → 上位レビュー
- **Day 14**: 取締役会付議

問題は **Day 1-10 の大部分が定型データ処理** だということ。「会計数字を取って → 前年比・予算比を計算して → 文章で説明する」という作業は、人間が書いても AI が書いても結論は同じ。なのに Day 14 までには経営判断の鮮度が失われています。

## 再設計 — 「書く」と「判断する」を分離する

<Principle num="01" heading="定型処理を AI に完全委譲し、人間は「判断」だけに集中する" />

従来 10 日かかっていた工程を、**AI が D+1 で初稿を出す → 人間が D+2 でレビューと判断のみ** に再配分。

**AI 担当 (定型処理):**
- 会計システムからのデータ取得 (MCP 経由)
- 予算マスタ・前年数値との突合
- 差異の事実記述 (数値の羅列レベル)
- 一次コメンタリー初稿 (定型フォーマットに沿った言語化)

**人間担当 (判断処理):**
- 差異要因の重要度評価 (どの数字を経営に持ち上げるか)
- 経営への示唆の言語化 (So What の抽出)
- 次月の注目論点の設定
- 取締役会への論点化・議題設計

## 実装 — Claude + MCP + Google Apps Script

### データ取得層 (MCP)
- **<Cite n={1}>freee MCP</Cite>** で会計 PL / BS を日次取得
- **Google Sheets MCP** で予算マスタ・過去実績・事業部別 KPI を読み込み
- **BigQuery MCP** で GA4 / 広告データを取得 (事業 KPI)

### 連結・差異分析層 (Python + GAS)
- 予算マスタの正本は **Sheets 1 枚** に集約 (分散させると差分管理が破綻)
- 差異計算は Sheets 数式ではなく **Python で実行 → 結果を Sheets に書き出す** (監査可能性)
- 予実対比は `(実績 - 予算) / 予算` ではなく **絶対額 + 率を並列** (経営層に読みやすい)

### コメンタリー生成層 (Claude)

一番効くのがここ。Claude に以下を与えて初稿生成:

- 過去 6 ヶ月分の取締役会コメント (文体学習)
- 今月の差異データ (構造化された数値と要因)
- プロンプト (フォーマット・禁止事項・重要度フラグ)

プロンプトの核は **「So What」必須 / 推測と事実を分離 / 感情語禁止 / 通期・単月影響分離 / 金額を分解**。これを `.claude/rules/board-comments.md` として永続化。

## 導入 ROI — 数字で見る効果

導入前と導入後の時間配分:

| 項目 | Before | After |
|------|--------|-------|
| 集計作業 | 80% | 20% |
| 判断・論点設計 | 20% | 80% |
| 月次締めから議論まで | 14 日 | 2 日 |
| 取締役会の議論の焦点 | 数字確認 | 判断と次アクション |

月額ランニングコスト:
- Claude API (Max Plan 一部): **$30-50/月**
- 会計 MCP サーバー (自前運用): **$0** (自社 VM 上で稼働)
- 合計: 月額 **¥5,000-8,000 程度**

**1 人分の工数 (月 1-2 営業日 = 時給換算 ¥80,000-160,000) が節約** できるなら、ROI は自明。

## 失敗事例 — 2026-04 のドラマ化インシデント

導入途中で大きな失敗もありました。広告の定期レビューで Claude が「17 倍の乖離！計測が壊れている可能性！」と警告してきたのですが、**実害を計算すると ¥3 だった** という事例。

原因: Claude の「役立ちたい」バイアスで、異常値を見た瞬間にドラマを構築する。計算と解釈を分離せず、解釈から入ってしまった。

対策として `.claude/rules/calibration.md` を追加し、全エージェントに強制適用:

- 異常値を見たら **先に金額/件数で実害を計算する** まで警告を出さない
- 「致命的」「破滅的」「壊れている」は使用禁止。代替は「実害 ¥X、優先度 P0」
- 仮説には **信頼度 (高/中/低) + 複数のデータソース裏付け** を付与

<PullQuote author="mixednuts Calibration Rule">
AI が生成した「異常値警告」の 9 割は、実害を計算すると無視していいレベル。地味な事実を地味に出すのが、意思決定の燃料になる。
</PullQuote>

この Calibration ルールが走るようになってから、FP&A の月次コメンタリーも「盛らない」トーンに安定。CFO / 取締役会にとって、月次資料で一番読みにくいのは **「書き手が盛っているか抑えているか」の判断コスト** なので、ここが安定するだけで読む側の負荷が激減します。

## FAQ

**Q. 会計データを AI に渡すとセキュリティ的に大丈夫か？**
A. Claude Enterprise 契約 (学習データに利用されないプラン) を使えばデータは安全。それでも機密性が高い場合は、数値を変換 (対数化・ハッシュ化) して渡す、または Azure OpenAI / Vertex AI の "中で閉じる" プランを使う選択肢あり。

**Q. どの会計システムなら MCP 連携しやすいか？**
A. API が素直なのは <Cite n={1}>freee</Cite> / 弥生クラウド / マネーフォワード。大企業の SAP / Oracle は個別実装コストがかかるが、FP&A 用の Data Mart を BigQuery / Snowflake に集約して、そこから MCP 経由で AI が読む構成が現実的。

**Q. 導入までの期間は？**
A. 標準的に **2 ヶ月**。1 ヶ月目 = 現状プロセス分析 + プロンプト設計、2 ヶ月目 = 並行運用 + 切替。3 ヶ月目以降は安定運用。

**Q. 導入後も経営企画の人員は必要か？**
A. 必要。むしろ "判断" に集中できる分、価値が上がる。削減されるのは作業時間であって、人員ではない。

**Q. 小規模企業 (売上 10 億未満) でも効果は出るか？**
A. 出る。むしろ経理 / 経営企画が 1-2 人の小規模企業の方が、1 人分の工数節約のインパクトが大きい。
